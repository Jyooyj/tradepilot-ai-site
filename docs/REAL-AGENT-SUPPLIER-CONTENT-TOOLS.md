# TradePilot Real Agent Supplier And Content Tools

## Scope

This round adds two real Agent tools:

- `generate_supplier_checklist`
- `generate_content_test_plan`

They are read-only planning tools. They do not contact suppliers, send messages, publish content, place orders, pay, save products, modify scores, overwrite records, or access external platforms.

## `generate_supplier_checklist`

This tool generates a copyable supplier communication checklist and message drafts.

It reuses the existing supplier communication module:

```text
src/utils/supplierCommunicationUtils.js
```

The server-side tool calls the same `generateSupplierCommunicationPack()` used by the frontend supplier panel, then reshapes the result for Agent trace display.

It covers:

- MOQ and small-batch trial quantity;
- wholesale price and tiered pricing;
- sample confirmation;
- replenishment cycle;
- material details;
- packaging;
- breakage and defects;
- after-sales policy;
- shipping time;
- whether small-batch trials are supported.

It never invents supplier name, phone, shop, certification, or delivery guarantee.

## `generate_content_test_plan`

This tool generates a first-round content testing plan.

It reads the canonical report packages already generated by the existing app:

- `result.xhsPackage`
- `result.douyinPackage`
- market evidence risk notes;
- compact memory summary.

It does not overwrite the original report content. It only produces an Agent-facing plan summary for trace and display.

It covers:

- 2-3 content angles;
- Xiaohongshu cover directions;
- Xiaohongshu title directions;
- Douyin first-three-second hooks;
- shooting tips;
- first-round test metrics;
- when to keep testing;
- when to optimize content;
- when to be cautious about replenishment.

## Planner Behavior

The LLM decides whether to call these tools through DashScope OpenAI-compatible Function Calling.

Planner guidance:

- profit/risk questions should prefer deterministic tools;
- history questions should use `retrieve_user_memory`;
- sampling, MOQ, packaging, delivery, and after-sales questions may use `generate_supplier_checklist`;
- first-round content validation questions may use `generate_content_test_plan`;
- do not call every tool automatically;
- call a tool only when it helps the current goal.

## Platform Data Safety

The content tool never accesses Xiaohongshu, Douyin, Taobao, or 1688.

It does not fabricate real plays, likes, saves, comments, inquiries, orders, sales, GMV, or heat metrics. It only suggests which metrics the user should manually observe during a real test.

## No Automatic Execution

The supplier tool produces copyable text only. The user decides whether to send it.

The content tool produces a plan only. The user decides whether to publish content.

This keeps the Agent semi-autonomous and controlled: it can recommend, but it cannot execute procurement, supplier communication, or platform publishing actions.

## Session Snapshot

Session snapshots may include compact summaries:

```text
supplierChecklistSummary
contentTestPlanSummary
```

The full product library and full planning payload are not stored as separate session top-level fields.

Old snapshots without these fields still resume normally.

## Current Limits

- Tools are deterministic and rule/template based.
- The content tool does not make a second LLM call; it uses existing report content packages.
- Supplier messages are drafts only and require human review.
- Platform metrics must be manually observed and entered by the user.
